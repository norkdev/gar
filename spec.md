# spec：Guided Agentic Retrieval for Literature Survey

このドキュメントは、対話の中で確定した補足・訂正・設計判断を取り込んだ **現時点の作業 spec** である。spec.md に書かれていない実装詳細（具体的なクラス構成・ライブラリ・関数 API など）は v1 実装の中で決める。

> 表向きラベルは "Guided Agentic Retrieval"。内部の設計用語としては従来通り「統治レイヤ／governance」を使う（4 要素を実装で見せるという主張を保つため）。

---

## 1. コンセプト

研究者・技術者の論文執筆前の **新規性・有用性の予備調査** を支援するエージェント。

- **公開情報**（外部 API 経由の論文等）を広く調査
- **未発表の自分のアイデア**（プライベートに保管）と対比
- 両者の関係（どの関連研究が近いか、どこが差分か）を **根拠付きで整理**

背骨は **Guided Agentic Retrieval**：エージェントが動的に retrieve するが、その全過程が統治レイヤ（governance）の下にある。

---

## 2. 譲れないコア

### (a) ツールは「判断しない」、材料を整える

- 新規性・有用性の **最終判断は人間（研究者）が下す**。エージェントは「最も近い関連研究はこれ、差分はここと思われる」を根拠付きで提示するに留める。
- 「新規性あり」と断定させない。研究者本人の責任で語るべき判断であり、AI の断定は危うい。
- この線引きにより HITL が「飾り」ではなく「必然」になる。

### (b) 公知と私案を厳密に分ける

- 新規性 =「公知の中に同じものが存在しないこと」。両者を混同したら議論が成立しない。
- 私案が外部に漏れれば、自分が温めているアイデアを自ら公知化してしまう。私案保護は当事者の利益として要る。
- データフロー上、両者は **別ロール・別ツール・別ストア** で隔離する。

### (c) 統治レイヤ 4 要素（実装で担保すべき機能要件）

1. **grounding 必須** — 全主張に出典を紐づける。公知文献の該当箇所、または web search の Text Fragment（後述）。引けないときは「引けない」と言わせ、捏造させない。
2. **HITL 承認（3 ゲート）** — (i) コンセプト確定後、検索前 / (ii) 検索結果の採用ソース選択 / (iii) 最終レポート生成前。
3. **監査ログ** — 全ツール呼び出し（何を・いつ・どのソースに）を構造化（JSONL）で記録、再現可能に。スキーマには `schema_version` を持たせる。
4. **RBAC** — 私案ツール群と公知ツール群を **別レジストリ** で持つ。私案パス未設定なら私案ツール群は agent に渡さない。認証・認可レイヤは枠だけ用意し v1 では pass-through。マルチユーザー化時に有効化。

## 3. ユーザー入力 / 私案ソース

### 入力形式

- **単一ファイル**：単一の Markdown
- **フォルダ**：Obsidian Vault またはそのサブフォルダ

### フォルダ走査ルール

- 対象拡張子：v1 は **Markdown のみ**（`.md`）
- 除外：
  - `.gitignore` を尊重
  - `.ignore`（後述）を尊重
  - `.obsidian/` などの Obsidian 固有システムフォルダはデフォルト除外

### 内容の扱い

- ユーザーのアイデアは通常まとまっておらず、ツール側で **コンセプトに要約** する。
- 要約結果は最初の HITL ゲート（コンセプト確定）でユーザーが確認・修正する。

---

## 4. 公開情報ソース

### 抽象化

- キーワード検索可能な API を **抽象インタフェース** の裏に置く。これにより別ソース（他の専門 DB 等）を後から差し替え・追加可能。

### v1 で実装

- **arXiv**：title + abstract + メタデータのみ取得（PDF 本文は取得しない）

### Future Work（抽象 IF の seam だけ用意、実装しない）

- PubMed / IEEE Xplore / Springer / Crossref など複数登録可能な構造
- 公知側 PDF 本文の取得・パース

### web search（補助）

- 別の独立ソースとして並列に持つ。**ソース名は `web search` で固定** し、ユーザーが登録した公開情報ソースと区別。
- grounding は **Text Fragments**（`#:~:text=...`）でハイライト位置を URL に内包 + 取得日時を必ず記録。

---

## 5. エージェント動作

### agent loop

- **retrieve → 推論 → 判断 →（再 retrieve or 停止）の反復**。LLM が次のツール使用を計画。
- シーケンシャルな固定パイプラインではなく、ユーザーの問いに柔軟に対応する構成。
- 全体の振る舞いはシステムプロンプトで制御する。

### retrieve 手法は「設計判断」として扱う

- semantic search / rerank / keyword search は **ループ内で使える複数のツールの一つ** として位置づける。
- v1 は **keyword 中心** で開始。semantic search / rerank は段階追加し、後で比較・検証できる位置に置いておく。

### 検索方針

- **導出コンセプト** だけでなく、**ユーザー原文の特徴的フレーズ** でも公開ソースを検索する（要約過程で消える特徴の取りこぼし防止）。

---

## 6. 評価軸

- **新規性** = 公開情報ソース内に同じコンセプトが見当たらない
- **有用性** = アイデアが何らかの便益を持つ（単なる新しさだけではない）
- 類似コンセプトは引用対象として提示
- 評価結果はあくまで材料、最終判断は人間

---

## 7. 最終レポート

### 保存場所・形式

- Markdown 形式
- 保存先：
  - ファイル指定時：そのファイルと同じフォルダ
  - フォルダ指定時：そのフォルダ
- ファイル名に **実行日** を含める。**同日複数実行は枝番** で区別。
- **過去レポートは上書きしない**。新規ファイルとして追加。

### 内容

- 導出されたコンセプト
- **参照した自分のアイデアファイル一覧**
- 公知の類似コンセプト（採用分の要約。複数なら共通点と各々の差分）
- 新規性・有用性の評価（grounding 付き）
- 発展 suggestion
- リファレンス（**採用 / 未採用** に分けてリスト）

---

## 8. 再実行の冪等性

- レポート保存フォルダに **`.ignore`**（`.gitignore` と同形式）を置く。
- 新規レポート生成のたびに **そのファイル名を追記**（行は増えていく）。
- 走査時は `.gitignore` と `.ignore` 両方を適用して既存レポートを対象外。
- `.gitignore` と分ける理由：ツール都合の除外をユーザーの git 境界に混ぜない。

## 9. アーキテクチャ

### 構成

- **フロントエンド**：Vite + React + TypeScript
  - 状態管理：TanStack Query（サーバ状態のみ、クライアント状態はほぼ要らない）
  - UI：Tailwind + shadcn/ui 程度の軽量構成
- **バックエンド**：FastAPI（Python 3.13 — Lambda マネージドランタイムに合わせて固定）
  - agent loop / tools / governance / 監査ログ / Anthropic SDK 呼出はすべてここ
- **通信**：HTTP（REST） + **SSE**（agent の進行ストリーム）
- **デプロイ（v1）**：
  - フロントエンド：S3 + CloudFront（静的ホスティング）
  - バックエンド：**Lambda + Function URLs**（Mangum 経由）
  - VPC は使わない（NAT Gateway 不要）
- **デプロイ（Phase 1+）**：必要に応じてバックエンドを ECS Fargate に移行

### AWS サービス構成

- **Step Functions**：エージェントオーケストレーションと HITL gate の wait-for-callback
- **DynamoDB**：エージェント状態 / チェックポイント / 監査メタデータ
- **S3**：監査ログ（JSONL） / 私案コンテンツ（暗号化） / 公開検索結果キャッシュ
- **Cognito**：認証（v1 は pass-through、Phase 1 で有効化）
- **KMS**：v1 はデフォルトキー、Phase 2 で per-tenant CMK 化
- **CloudWatch**：実行メトリクス、Lambda ログ

### HITL ゲートの実装

- ゲートは **durable state**（コルーチンの await ではない）：
  - バックエンドが state = `awaiting_user` に入って停止
  - フロントエンドが状態を購読 / ポーリングして UI を出す
  - ユーザー入力 → Step Functions に task token を返して再開
- これにより HITL 待機中の compute コストはゼロ。

## 10. v1 から仕込む「スケール seam」（7 点）

エンタープライズ規模への拡張を後の rewrite なしで可能にするため、v1 から以下を規約として組み込む。コード量はほぼ増えない。

1. **`tenant_id` を全データ・全ログに必ず付与**。v1 は常に `default`。フィールドだけ持つ。
2. **UI → AWS の直接呼び出し禁止**。フロントは必ず自前 API（FastAPI）経由。
3. **エージェント状態は DynamoDB に同期**（フロントの状態だけに頼らない）。
4. **HITL ゲートは durable state**（前項の通り）。
5. **LLM クライアントを抽象化**。Anthropic ↔ Bedrock を 1 行で差し替え可能に。
6. **監査ログに `schema_version`** フィールドを最初から。
7. **認可チェックを API 境界で実施**（v1 では常に許可するスタブだが、判定箇所だけ作っておく）。

## 11. v1 スコープ確定事項（実装する / しない）

### 実装する

- 私案ソース：Markdown のみ
- 公知ソース：arXiv（abstract + メタデータのみ）+ web search（Text Fragments grounding）
- agent loop + 3 つの HITL ゲート
- 監査ログ（JSONL on S3、`schema_version` 付き）
- レポート生成（実行日 + 枝番、上書きしない、`.ignore` で再実行除外）
- スケール seam 7 点
- AWS バックエンド（Lambda + Step Functions + DynamoDB + S3）
- React + TypeScript フロントエンド

### 実装しない（Future Work）

- 私案・公知両側の **PDF 本文取得・パース**（両側同時に拡張する想定）
- 画像処理
- 公開情報ソースの追加（PubMed / IEEE / Springer / Crossref など、その他の専門 DB を含む）
- マルチユーザー実運用（認証・認可は seam のみ）
- per-tenant CMK / multi-region / VPC endpoint
- Bedrock 経由 LLM 切替（compliance 要件が立ったタイミングで）

## 12. 制約・ガード

- **個人プロジェクト**。現職の資産・社内コード・顧客データ・社内ノウハウを一切混入させない。使うのは公開 API と自分でゼロから書くコードのみ。
- 公開 API の **利用規約・レート制限を尊重**。クレデンシャルは安全管理、リポジトリに含めない（`.gitignore` 整備が前提）。
- 公開リポジトリ前提：**README で設計判断を語れる構造** にしておく（なぜ agent loop か／なぜ判断せず材料準備か／なぜ grounding 必須か／統治 4 要素の実装／retrieve 手法の扱い／限界と Future Work／スケール seam の意図）。

---

## 13. フェーズ進行（参考）

- **Phase 0（v1）**：個人用途。React + FastAPI（Lambda）+ AWS バックエンド。tenant 常に `default`。
- **Phase 1（小規模チーム）**：Cognito 認証 ON、複数 tenant、必要なら FastAPI を ECS Fargate に移行。
- **Phase 2（中〜大規模）**：Bedrock 経由 LLM、per-tenant CMK、OpenSearch で監査クエリ、UI の本格化。
- **Phase 3（大規模 / regulated）**：multi-region、VPC endpoint / PrivateLink、CloudHSM、テナント別 IdP federation。

---

## 14. 評価できる余地（実装段階で守る構造）

- 出力の grounding が信頼できるか（出典が実在し、主張を支持しているか）を後で検証できる構造にしておく。
- 失敗モード（捏造された関連研究、私案漏洩の試み、根拠不足の断定など）を後で洗い出し、統治レイヤがそれを防げているかを確認できる構造にしておく。
- 評価の作り込み自体は別フェーズ。実装段階では「評価しやすい構造」にしておけば十分。具体的には：
  - 監査ログを構造化（JSONL）で書く
  - retrieve 手法を切替可能にしておく
  - HITL ゲートでのユーザー判断もログに残す

## 15. MCP サーバー（v1.1）

v1.0 の二つのクライアント（CLI / Web UI）に、MCP クライアント（Claude Code /
Claude Desktop）から GAR を操作できる三つ目のクライアント面を追加した。詳細な
設計判断は `plan.md`（Phase 1）を正とし、ここでは spec として要点のみ確定する。

- **公開面はゲート、内部ツールではない**：MCP が公開するのはラン管理と 3 つの
  HITL ゲート（`start_survey` / `list_runs` / `get_run_status` /
  `review_concept` / `select_sources` / `approve_report` / `get_report`）であり、
  `search_arxiv` のような低レベル検索は公開しない。低レベルツールを公開すると
  MCP クライアント側 LLM がエージェントループを代替でき、grounding・HITL・監査
  をバイパスしたサーベイが成立する。ゲートを境界にすることで統治レイヤは
  プロトコル境界を越えても不変（＝「統治付きサブエージェント」）。
- **薄い REST クライアントとして実装**：MCP サーバーは `gar_backend` を import
  せず、Web UI と同じ HTTP API を叩く（base URL = `GAR_API_URL`）。これにより
  スケール seam #2（UI は AWS を直接呼ばない）が MCP にも及び、AWS 移行後も
  base URL と認証ヘッダの差し替えだけで同じサーバーが動く。
- **RBAC は構造的不在で再現**：起動ロール `GAR_MCP_ROLE`（既定 `public`）に応じ、
  上位ロール専用ツールはスキーマに現れない（呼び出し時拒否ではない）。v1.1 は
  公開ツールのみ。ideas（私案）検索を MCP に足すのは Future Work。
- **MCP 操作も監査**：全リクエストに `X-GAR-Client: mcp` を付与し、監査レコードの
  `client` フィールドに記録する（`schema_version` を 1.1 に）。
- **ノート入力は content-upload パスのみ**：MCP は `vault_path` を公開せず、
  ノート内容を引数で受ける。リモートバックエンドでも同一挙動になるため。
  レポートの保存はクライアント側の責務。
- **同期だが将来のポーリングを見越したスキーマ**：v1.1 は同期呼び出しだが、
  `start_survey` は `run_id` を返し `get_run_status` でポーリングする形にして
  あるので、Phase 2 で API が 202 + ポーリングに変わっても MCP ツールの形は
  変わらない。
